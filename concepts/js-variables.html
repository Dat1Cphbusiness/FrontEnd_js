const · let · var

// JavaScript variable declarations — visually compared

01 — Overview
const
block-scoped not reassignable no hoisting surprise

Declares a binding that cannot be reassigned. The value of primitives is locked. Objects/arrays declared with const can still have their contents mutated.

let
block-scoped reassignable no hoisting surprise

Declares a reassignable binding limited to the enclosing block ({}). The modern default when a variable needs to change.

var
function-scoped reassignable hoisted + initialized

The legacy declaration. Leaks out of blocks into the enclosing function (or global). Hoisted and initialized to undefined — a common source of bugs.

04 — Code Examples
⬡ const — immutable binding
const score = 42;
score = 50; // ✗ TypeError!

// But objects CAN be mutated:
const player = { name: "Ada" };
player.name = "Grace"; // ✓ OK
player = {}; // ✗ TypeError!
◈ let — reassignable, block-scoped
let count = 0;
count = 1; // ✓ OK

if (true) {
  let inner = "block only";
}
console.log(inner);
// ✗ ReferenceError!
◎ var — function-scoped, leaks
if (true) {
  var surprise = "I leaked!";
}
console.log(surprise);
// ✓ "I leaked!" — var escapes

for (var i = 0; i < 3; i++) {}
console.log(i); // 3 — leaks!
05 — Hoisting

var — hoisted & initialized

Declaration is moved to the top of the function, initialized to undefined. You can read a var before its line — you just get undefined.

console.log(x); // undefined (not error!)
var x = 5;
console.log(x); // 5

let — hoisted into TDZ

Hoisted but not initialized. Accessing before declaration throws a ReferenceError. This "Temporal Dead Zone" protects you.

console.log(y); // ✗ ReferenceError
let y = 5;
console.log(y); // 5

const — hoisted into TDZ

Same as let — in the TDZ until its line. Must be initialized immediately; no separate declaration and assignment.

console.log(z); // ✗ ReferenceError
const z = 5;
console.log(z); // 5
